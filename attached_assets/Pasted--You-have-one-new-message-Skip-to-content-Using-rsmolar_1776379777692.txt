## My recommendation: **Option C** (fresh random key, same pattern as production but not the same value)

Reasoning:

**Against Option A (same as VPS):** dev/prod sharing an encryption key is a security anti-pattern. If your Replit project ever leaks — screen recording, logs, exposed `.env`, stale session on a shared machine, someone else getting Replit access — the production key leaks with it. Anyone who gets the Replit key can decrypt every secure-vault entry on the live site. The whole point of a dedicated `APP_ENCRYPTION_KEY` env var is that each deploy target has its own. Don't collapse that isolation.

**Against Option B (use DATABASE_URL as the value):** it "works" in the sense that existing Replit dev blobs remain decryptable, but the value looks like a connection string (weird), and it silently couples dev's encryption to a value that's also the DB credential. If someone later rotates the DATABASE_URL in Replit thinking it's just a connection-string change, they lose any encrypted data that was migrated under the old URL. This is exactly the footgun we just removed from production — don't reintroduce it in dev.

**For Option C (fresh random):** clean separation, no confusion. The only cost is that existing Replit dev blobs (any vault entries, SSH credentials, gateway secrets in the dev DB) won't decrypt under the new key. For a dev environment, that's usually fine — dev data is typically throwaway. If you've accumulated real dev test data you care about, tell me and we'll do a one-time migration, but in most cases the cleanest move is to just truncate the affected tables.

# Guardian-Layer — developer context

## Stack
- pnpm monorepo. Workspaces: `artifacts/api-server`, `artifacts/guardianlayer`, `lib/db`, `lib/api-zod`, `lib/api-client-react`, `lib/integrations-*`.

- `PORT`, `DATABASE_URL`, `NODE_ENV`, `JWT_SECRET`
- `APP_ENCRYPTION_KEY` — rotating this after data is written causes silent decrypt failures in secure-vault, gateway, remote-machines, node-diagnostics, tailscale-imported devices. Not recoverable without re-encrypting the rows under the old key first.

## Auth + routing (post-patch)
- `authMiddleware.isPublicPath()` only gates `/api/*` paths. SPA and static assets pass through to `express.static`.
- Public `/api` endpoints are listed in `PUBLIC_PATHS` at the top of `src/middleware/auth.ts`: login, firebase, webauthn/login/*, medinvest/*, did/*, health, healthz, /metrics.
- IP guard auto-allows 127.0.0.1 + RFC1918 ranges. External IPs get rate-tracked; abusive ones get blocked with escalating durations.
- Global rate limiter: 1000 req / 15 min per IP.

## Encryption (post-patch)
Six files use AES-256-GCM with a key derived from `sha256(APP_ENCRYPTION_KEY)`: `lib/seed-remote-machines.ts`, `lib/seed-tailscale.ts`, `routes/gateway.ts`, `routes/node-diagnostics.ts`, `routes/remote-maintenance.ts`, `routes/secure-vault.ts`. ```
`push` is interactive — say YES to additive changes (ADD column, CREATE table, ADD constraint), STOP and ask a human before any DROP, TRUNCATE, or RENAME.

## Step 2f — Connect Replit to GitHub and push

Two options depending on your Replit UI:

**Option A — Replit's built-in "Connect to GitHub" button (preferred):**
1. Open the Version Control / Git pane in Replit (left sidebar icon).
2. Click "Connect to GitHub" or "Publish to GitHub" (button text varies).
3. Authorize Replit → GitHub if first time.
4. When prompted, select the existing repo you just created (or allow Replit to create a new one).
5. Click "Push".

**Option B — Manual via Replit shell (works if A is flaky):**
Open the Shell in Replit and run:
```
git remote add origin git@github.com:YOUR-USERNAME/guardian-layer.git
git branch -M main
git push -u origin main
```
Replit's shell should have SSH keys already set up for GitHub. If it asks for auth and fails, use HTTPS with a Personal Access Token:
```
git remote add origin https://github.com/YOUR-USERNAME/guardian-layer.git
git push -u origin main
```
On push, it'll prompt for username (your GitHub username) and password (use a GitHub Personal Access Token, NOT your account password — create one at github.com/settings/tokens, scope: `repo`).
